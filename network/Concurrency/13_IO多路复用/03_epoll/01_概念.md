# IO多路复用：epoll 学习笔记

## 一、 概述

`epoll`（全称 eventpoll）是 Linux 内核实现 IO 多路转接/复用的一个重要机制。IO 多路转接允许在一个操作里同时监听多个输入输出源，当其中一个或多个源可用时返回，然后再对其进行读写操作。

作为 `select` 和 `poll` 的升级版，`epoll` 极大地改进了工作方式，因此具有更高的运行效率。特别是在**多路复用的文件数量庞大、IO流量频繁**的场景下，`select` 和 `poll` 表现较差，强烈推荐使用 `epoll`。

### epoll 与 select/poll 的核心优势对比：

1. **底层数据结构**：`select` 和 `poll` 针对待检测集合采用**线性轮询**方式处理；而 `epoll` 是基于**红黑树**来管理待检测集合的，增删改查效率极高。
2. **检测机制**：`select` 和 `poll` 每次都会线性扫描整个集合，集合越大速度越慢；`epoll` 采用**回调机制**，处理效率不会随着检测集合的变大而下降。
3. **内存拷贝**：`select` 和 `poll` 在工作时存在内核空间与用户空间之间数据的频繁拷贝问题；`epoll` 中内核区和用户区使用的是**共享内存**（基于 mmap 内存映射区实现），省去了不必要的内存拷贝开销。
4. **就绪结果获取**：`select` 和 `poll` 返回后，需要程序再次遍历集合才能知道哪些文件描述符（FD）是就绪的；`epoll` 返回时会**直接提供已就绪的文件描述符集合**，无需进行二次遍历检测。
5. **连接数限制**：`epoll` 没有最大文件描述符的硬性限制，仅受系统中进程能打开的最大文件数目限制。

## 二、 操作函数详解

`select` 和 `poll` 低效的原因之一，是将“添加和维护待检测任务”与“阻塞进程/线程”这两个步骤合二为一。而大多数应用场景中，需要监视的集合是相对固定的，不需要每次都修改。
`epoll` 将这两个操作彻底分开：**先用 `epoll_ctl()` 维护等待队列，再调用 `epoll_wait()` 阻塞进程。**

`epoll` 一共提供了三个 API 函数（需引入头文件 `#include <sys/epoll.h>`）：

### 1. 创建 epoll 实例
```c
int epoll_create(int size);
```
* **功能**：在内核区创建一棵用来管理文件描述符的红黑树，以及一个维护就绪状态的链表。
* **参数 `size`**：传入一个大于 0 的整数。在早期的 Linux 内核中，这个参数用来告诉内核需要管理的句柄的大致数量。但在 Linux 2.6.8 之后，红黑树已经是动态分配的了，这个参数已经被完全忽略，但为了向前兼容，必须传入一个 **大于 0** 的值（例如传 1 或 100 都可以）。
* **返回值**：成功时，返回一个非负的文件描述符（`epfd`），它是后续操作这棵红黑树的唯一凭证；失败返回 `-1`。

### 2. 管理红黑树上的文件描述符
```c
int epoll_ctl(int epfd, int op, int fd, struct epoll_event *event);
```
* **功能**：对 epoll 树上的文件描述符执行添加、修改或删除操作。
* **参数 `epfd`**：由 `epoll_create` 返回的 epoll 实例句柄。
* **参数 `op`**：要执行的具体动作，取值如下：
  * `EPOLL_CTL_ADD`：往 epoll 树上**添加**一个新的文件描述符 `fd`。
  * `EPOLL_CTL_MOD`：**修改**已经挂在树上的 `fd` 的监听事件。
  * `EPOLL_CTL_DEL`：将 `fd` 从 epoll 树上**删除**，不再监控。
* **参数 `fd`**：你需要委托内核监控的目标文件描述符（如监听套接字或通信套接字）。
* **参数 `event`**：指向 `epoll_event` 结构体的指针，用于告诉内核你要监听该 `fd` 的什么事件（例如 `EPOLLIN` 读事件）。当 `op` 为 `EPOLL_CTL_DEL`（删除）时，这个参数通常传 `NULL`。
* **返回值**：成功返回 `0`；失败返回 `-1`。

### 3. 检测并获取就绪的事件
```c
int epoll_wait(int epfd, struct epoll_event* events, int maxevents, int timeout);
```
* **功能**：阻塞监听 epoll 树，当有文件描述符就绪（或超时）时返回，并将就绪的事件打包传出。
* **参数 `epfd`**：由 `epoll_create` 返回的 epoll 实例句柄。
* **参数 `events`（传出参数）**：这是一个由用户分配好的 `epoll_event` 结构体数组。内核会把所有触发了的事件以及对应的文件描述符拷贝到这个数组里，程序只需遍历这个数组即可处理所有业务。
* **参数 `maxevents`**：修饰 `events` 数组，告诉内核这个数组最多能装多少个事件（通常写数组的容量）。
* **参数 `timeout`**：设置函数的阻塞等待时间，单位为**毫秒（ms）**：
  * `timeout = -1`：**一直阻塞**，直到至少有一个文件描述符状态发生变化才返回。
  * `timeout = 0`：**非阻塞**，立刻检查就绪链表，无论有没有事件都马上返回。
  * `timeout > 0`：**限时阻塞**，最多等待 `timeout` 毫秒。如果期间有事件发生则提前返回，如果一直没动静则超时返回 `0`。
* **返回值**：
  * `> 0`：成功，返回实际就绪的文件描述符总数（也是 `events` 数组中前几个元素有实际数据的范围）。
  * `== 0`：超时，没有事件发生。
  * `-1`：调用失败。

## 三、 epoll 的使用流程

使用 `epoll` 搭建并发服务器的标准操作步骤如下：

```c
// 1. 创建监听的套接字
int lfd = socket(AF_INET, SOCK_STREAM, 0);

// 2. 设置端口复用 (可选，防止 TIME_WAIT 导致端口被占用)
int opt = 1;
setsockopt(lfd, SOL_SOCKET, SO_REUSEADDR, &opt, sizeof(opt));

// 3. 使用本地的 IP 与端口和监听的套接字进行绑定
struct sockaddr_in serv_addr;
// ... 初始化 serv_addr 结构体 ...
int ret = bind(lfd, (struct sockaddr*)&serv_addr, sizeof(serv_addr));

// 4. 给监听的套接字设置监听
listen(lfd, 128);

// 5. 创建 epoll 实例对象
int epfd = epoll_create(100);

// 6. 将用于监听的套接字 (lfd) 添加到 epoll 实例中
struct epoll_event ev;
ev.events = EPOLLIN;    // 委托监听读事件
ev.data.fd = lfd;       // 记录对应的文件描述符
epoll_ctl(epfd, EPOLL_CTL_ADD, lfd, &ev);

// 定义一个数组用于接收内核传出的就绪事件
struct epoll_event evs[1024];
int size = sizeof(evs) / sizeof(struct epoll_event);

// 7. 进入死循环，持续检测并处理就绪的文件描述符
while (1) {
    // 阻塞等待事件发生
    int num = epoll_wait(epfd, evs, size, -1);
    
    // num 是实际发生事件的数量，直接遍历这 num 个元素即可
    for (int i = 0; i < num; ++i) {
        int curfd = evs[i].data.fd;
        
        // 情况 A: 如果是监听的文件描述符就绪，说明有新客户端发起连接
        if (curfd == lfd) {
            int cfd = accept(curfd, NULL, NULL);
            // 将新得到的文件描述符添加到 epoll 实例中监听
            ev.events = EPOLLIN;
            ev.data.fd = cfd;
            epoll_ctl(epfd, EPOLL_CTL_ADD, cfd, &ev);
        }
        // 情况 B: 如果是通信的文件描述符就绪，说明客户端发来了数据
        else {
            char buf[1024] = {0};
            int len = recv(curfd, buf, sizeof(buf), 0);
            
            if (len == 0) {
                // 客户端已经断开连接
                // 将该文件描述符从 epoll 模型中删除
                epoll_ctl(epfd, EPOLL_CTL_DEL, curfd, NULL);
                close(curfd);
            } 
            else if (len > 0) {
                // 正常接收到数据，进行处理（这里以回发 echo 为例）
                send(curfd, buf, len, 0);
            }
            else {
                // 发生错误 (需处理异常)
                perror("recv");
            }
        }
    }
}
```

## 四、epoll的工作模式
# 4.1 水平模式

水平模式可以简称 **LT 模式** ，LT（Level Triggered）是 epoll **默认（缺省）** 的工作方式。它同时支持阻塞（block）和非阻塞（no-block）socket。在这种做法中，内核会持续通知使用者哪些文件描述符已经就绪。**只要条件满足，内核就会一直提醒你**，直到你完成了相应的 IO 操作。

### 水平模式的特点

**① 读事件**：如果文件描述符对应的读缓冲区还有数据，读事件就会被触发，`epoll_wait()` 会解除阻塞。
* 当读事件被触发，`epoll_wait()` 解除阻塞，之后就可以接收数据了。
* **【核心特性】** 如果接收数据的 `buf` 很小，一次读取没能把缓冲区数据全部读出，那么在下一次调用 `epoll_wait()` 时，**读事件会继续被触发**，直到数据被全部读出为止。
* 因为读数据是被动的（不知道对方何时发数据），通过读事件的持续检测，可以确保数据不会由于单次处理不完而丢失。

**② 写事件**：如果文件描述符对应的写缓冲区**可写**（即缓冲区不满），写事件就会被触发，`epoll_wait()` 会解除阻塞。
* 当写事件被触发，`epoll_wait()` 解除阻塞，之后就可以将数据写入到写缓冲区了。
* **【核心特性】** 只要写缓冲区没有被写满，写事件就会**一直被触发**。
* 因为写数据是主动的，且通常情况下写缓冲区都是不满的（处于可写状态），因此如果一直开启写事件监听，`epoll_wait` 会频繁返回，导致 CPU 占用率升高。所以通常**不需要持久监听写事件**。

---

### 📦 辅助理解：快递驿站比喻（LT 唠叨版）

假设快递驿站（内核）里新到了你的 10 个包裹（缓冲区里的数据）。

* **LT 模式下的驿站老板：**
  老板是个非常尽职但有点“唠叨”的人。他看到你有包裹到了，立刻给你打了个电话：“你有包裹，快来拿！”（**触发读事件**）
  
  你骑着小单车去驿站，篮子太小（**`buf` 小**），只装了 3 个包裹回家。
  
  当你回到家还没坐稳，老板看了一眼仓库，发现还有你的 7 个包裹，于是**立刻又给你打了个电话**：“喂，你那还有 7 个包裹没拿呢，快来拿！”
  
  如果你还是只拿 3 个，老板就会**一遍又一遍地给你打电话**，直到你把所有包裹都搬空。

* **评价：**
  虽然老板很唠叨，但**非常安全**。只要你没拿完，他绝对不会忘了提醒你，你永远不用担心漏掉任何一个快递。

### 🛠️ LT 模式使用建议
1. **编程门槛低**：对逻辑严谨性要求不如 ET 模式那么高，适合初学者，不容易出现“数据读一半卡死”的 Bug。
2. **性能损耗**：由于只要有数据就会反复触发，当并发量极大时，内核频繁唤醒用户进程会产生一定的性能开销。
3. **配合阻塞 IO**：LT 模式通常可以配合阻塞 IO 使用，因为即便 read 阻塞了一会儿，下次事件还会触发。

# 4.2 边缘模式

边缘模式可以简称 **ET 模式**，ET（Edge Triggered）是 epoll 的**高效**工作方式。与 LT 模式不同，**ET 模式强烈建议（甚至可以说必须）配合非阻塞（no-block）socket 使用**。在这种做法中，内核只在文件描述符状态**发生变化**的那一瞬间通知使用者一次。如果我们没有把数据处理完，内核**绝对不会**再次通知使用者，直到下一次有新状态发生。

### 边缘模式的特点

**① 读事件**：如果文件描述符对应的读缓冲区有**新数据到达**（状态从空变成有，或者从少变成多），读事件就会被触发，`epoll_wait()` 解除阻塞。
* 当读事件被触发，`epoll_wait()` 解除阻塞，之后就可以接收数据了。
* **【核心区别】** 如果接收数据的 `buf` 很小，不能一次性将缓冲区数据读出，那么**未读完的数据不会再次触发读事件**（除非对方又发了新的数据过来）。
* 因此，在 ET 模式下处理读事件，必须**在一个循环中不断调用 `read/recv`**，直到把缓冲区的数据全部读空（通常表现为 `read` 返回 `-1` 且错误码为 `EAGAIN` 或 `EWOULDBLOCK`），才能退出循环。
* 因为内核不会反复发送通知，极大地减少了 `epoll_wait()` 被唤醒的次数，系统整体效率大幅提升。

**② 写事件**：如果文件描述符对应的写缓冲区状态发生改变（通常是从**满（不可写）变成了不满（可写）**），写事件就会被触发一次，`epoll_wait()` 解除阻塞。
* 当写事件被触发，`epoll_wait()` 解除阻塞，之后就可以将数据连续写入到写缓冲区了。
* 同样地，你必须在循环中不断调用 `write/send`，直到写缓冲区被再次塞满（返回 `EAGAIN` 错误）为止。
* 在 ET 模式下，写事件的通知是非常“吝啬”的，只有在缓冲区腾出空间的那一刻才会通知一次，不会像 LT 模式那样只要有空位就一直通知。

### 📦 辅助理解：快递驿站比喻（ET 高冷版）

假设快递驿站（内核）里新到了你的 10 个包裹（缓冲区里的数据）。

* **ET 模式下的驿站老板：**
  老板是个非常高冷、讲究效率的人。他看到你的包裹到了，给你发了一条微信：“有包裹，速来，**只通知一次**。”（**触发一次读事件**）
  
  你骑着小单车去驿站，篮子太小（**`buf` 小**），只装了 3 个包裹回家。
  
  当你回到家后，驿站里虽然还有你的 7 个包裹，但高冷老板**绝对不会**再给你发微信了。
  如果你不主动回去拿，这 7 个包裹会永远死锁在驿站里。直到下个月，你又网购了**新的包裹**到达驿站，老板才会再次给你发一条微信。

* **正确的拿快递（写代码）姿势：**
  收到老板微信后，你必须自己带个大卡车，或者骑着小单车**疯狂来回跑（开启 `while` 循环）**。直到你走进驿站，老板对你说：“别找了，你的包裹已经全部拿光了（**返回 `EAGAIN`**）”，你才可以安心回家睡觉，等待老板的下一次微信。

### ⚠️ ET 模式避坑指南（面试常考）
为什么 ET 模式**必须**用非阻塞（`O_NONBLOCK`）的 Socket？
因为在 ET 模式下，你必须用 `while` 循环去读数据直到读空。如果是**阻塞**的 Socket，当缓冲区的数据被读完时，最后一次 `read` 动作会把整个程序卡死（阻塞）在那里，永远等不到下一个事件了。而非阻塞 Socket 在读空时会立刻报错 `EAGAIN` 退出循环，让程序继续运行。
