# 🐳 Dockerfile 核心技术与实战笔记

---

## 1. Dockerfile 基本概念

Dockerfile 是用来构建 Docker 镜像的文本文件，是由一条条构建镜像所需的**指令和参数**构成的脚本。

### 1.1 为什么不直接用 `docker commit`？
虽然可以通过 `docker commit` 将一个容器手动打包成新镜像，但在微服务架构中，服务往往需要配置很多动态参数，并且要与各种中间件建立网络联系。此时手动 `commit` 既不方便也难以追溯和自动化。使用文本化的 Dockerfile 脚本则能完美解决环境配置、自动化构建的问题。

### 1.2 Docker 体系的“三剑客”
* **Dockerfile**：面向**开发**。是软件的“原材料”，定义了镜像需要的一切东西（包括执行代码、环境变量、依赖包、运行时环境、操作系统发行版等）。
* **Docker 镜像**：面向**交付**。是通过 `docker build` 消费 Dockerfile 后产生的“交付标准”。
* **Docker 容器**：面向**部署与运维**。是软件镜像的运行态，即依照镜像运行的实体实例。
*(三者缺一不可，合力充当 Docker 体系的基石)*

## 2. 基础规范与执行引擎流程

### 2.1 Dockerfile 基础语法规范
1. 每条**保留字指令**都必须为**大写字母**，且后面要跟随至少一个参数。
2. 指令按照**从上到下**的顺序执行。
3. `#` 符号表示注释。
4. 每条指令都会**创建一个新的镜像层**并对镜像进行提交。

### 2.2 Docker 引擎执行流程（套娃机制）
Docker 执行 Dockerfile 的大致底层流程如下：
1. Docker 从基础镜像运行一个**初始容器**。
2. 执行一条指令并对容器**做出修改**。
3. 执行类似 `docker commit` 的操作，提交生成一个**新的镜像层**。
4. Docker 基于刚提交的新镜像，运行一个**新容器**。
5. 循环执行下一条指令，直到所有指令都执行完成。

## 3. 常用保留字指令大全 (关键字)

这些大写关键字是编写 Dockerfile 的核心骨架：

| 保留字 | 核心作用 | 详细解析与区别 |
| :--- | :--- | :--- |
| **`FROM`** | **指定基础镜像** | 当前新镜像是基于哪个镜像建立的。**必须是 Dockerfile 的第一条有效指令**。 |
| **`MAINTAINER`** | **镜像维护者** | 描述镜像维护者的姓名和邮箱地址。 |
| **`RUN`** | **构建时执行** | 镜像**构建 (build) 时**需要运行的命令。<br>• **Shell 格式**：等同于终端命令（如 `RUN apt-get install vim`）。<br>• **Exec 格式**：`RUN ["可执行文件", "参数1", "参数2"]` |
| **`EXPOSE`** | **暴露端口号** | 声明该容器对外暴露的端口号。 |
| **`WORKDIR`** | **指定工作目录** | 指定在创建容器后，终端默认登录进来的“落脚点”工作目录。 |
| **`USER`** | **指定运行用户** | 指定该镜像以什么样的系统用户去执行（默认是 root）。 |
| **`ENV`** | **设置环境变量** | 描述在构建的过程中用到的环境变量，后续指令可以直接引用它。 |
| **`VOLUME`** | **挂载数据卷** | 自动创建容器数据卷，用于数据保存和持久化工作。 |
| **`ADD`** | **高级拷贝** | 将宿主机目录下的文件拷贝进镜像。**自带黑科技：会自动处理 URL 并且解压 tar 压缩包。** |
| **`COPY`** | **普通拷贝** | 类似于 ADD，但只是纯粹拷贝文件和目录到镜像中，没有自动解压功能。 |
| **`CMD`** | **运行时执行** | 指定**容器启动后 (run)** 要干的事情。支持 shell 和 exec 格式。<br>⚠️ **注意**：可以有多个 CMD 指令，但**只有最后一个会生效**；并且它**会被 `docker run` 命令行后的参数覆盖**。 |
| **`ENTRYPOINT`** | **追加式运行** | 也是指定容器启动时要运行的命令。类似于 CMD，但 **不会被 `docker run` 后的命令覆盖**，反而 `docker run` 后的命令行参数会被当做参数，**追加**给 `ENTRYPOINT` 指定的程序。 |

## 4. 虚悬镜像 (Dangling Image)

**什么是虚悬镜像？**
镜像名称 (Repository) 和标签 (Tag) 都显示为 `<none>` 的镜像就是虚悬镜像。当我们重构镜像时，新的镜像占用了原有的名字，旧镜像就会变成“虚悬”状态。这种镜像没有实际价值，纯粹浪费磁盘空间，一般需要将其删除。

**虚悬镜像清理指令：**
* **查看所有的虚悬镜像：**
  ```bash
  docker image ls -f dangling=true
  ```
* **一键删除所有的虚悬镜像：**
  ```bash
  docker image prune
  ```
