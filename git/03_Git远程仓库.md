# Git 远程仓库与协作开发笔记

## 1. 常用的远程仓库托管平台
* **GitHub**：全球最大的面向开源及私有软件项目的托管平台。因为只支持 Git 作为唯一的版本库格式进行托管，所以叫 GitHub。
* **码云 (Gitee)**：国内优秀的代码托管平台。相比于 GitHub，由于服务器在国内，国内用户的网络访问和代码同步速度会更快。
* **GitLab**：一个用于代码仓库管理的开源项目。它使用 Git 作为代码管理工具，并在此基础上搭建 Web 服务。一般常用于企业、学校等内部局域网搭建 Git 私服，以保证代码的安全性和私密性。

## 2. 配置 SSH 公钥（免密通讯）
配置 SSH 公钥可以建立本地电脑与远程仓库之间的安全信任关系，实现免密推送和拉取代码。

**完整操作步骤**：
1. **生成 SSH 密钥对**：打开 Git Bash，输入以下命令（将其中的邮箱替换为您自己的邮箱），然后连续按三次次回车即可生成：
   ```bash
   ssh-keygen -t rsa -C "your_email@example.com"
   ```
2. **查看并复制公钥**：输入以下命令在终端打印公钥内容，将以 `ssh-rsa` 开头的整段字符全部复制：
   ```bash
   cat ~/.ssh/id_rsa.pub
   ```
3. **添加到远程仓库**：登录 GitHub 或 Gitee，进入 **Settings (设置)** -> **SSH and GPG keys (SSH 公钥)** -> 点击 **New SSH key (添加公钥)**，将复制的内容粘贴并保存。
4. **测试连通性**：在 Git Bash 中输入以下命令，若提示 `successfully authenticated` 即代表配置成功：
   ```bash
   ssh -T git@github.com   # 如果使用的是 GitHub
   # 或者 ssh -T git@gitee.com # 如果使用的是码云
   ```

## 3. 远程仓库基础操作

### 3.1 添加与查看远程仓库
* **添加远程仓库关联**：将本地仓库与一个远程仓库地址绑定。
  ```bash
  git remote add <远端名称> <仓库路径>
  # 示例：git remote add origin git@github.com:xxx/xxx.git
  ```
* **查看是否有远程仓库**：
  ```bash
  git remote
  # 加 -v 参数可查看详细的抓取和推送地址：git remote -v
  ```

### 3.2 推送代码到远程仓库 (Push)
* **命令形式**：
  ```bash
  git push [-f] [--set-upstream] <远端名称> <本地分支名>
  ```
* **参数解析**：
  * `-f` (force)：强制推送，用本地代码强制覆盖远程仓库（危险操作，团队协作中谨慎使用）。
  * `--set-upstream` (或 `-u`)：推送到远端的同时，建立起本地分支和远端分支的追踪/关联关系。绑定后，下次在此分支推送只需输入 `git push`。

## 4. 获取远程仓库代码

### 4.1 从远程仓库克隆 (Clone)
将远程仓库完整的代码和历史记录克隆到本地，通常用于参与一个新项目。
* **命令**：
  ```bash
  git clone <仓库路径> [本地目录]
  ```

### 4.2 抓取 (Fetch) vs 拉取 (Pull)
* **抓取 (Fetch)**：`git fetch [remote name] [branch name]`
  * 作用：将仓库里的更新都抓取到本地，但**不会自动进行合并 (merge)**。
  * 细节：如果不指定远端名称和分支名，就会抓取所有分支。
* **拉取 (Pull)**：`git pull [remote name] [branch name]`
  * 作用：将远端仓库的修改拉取到本地并**自动进行合并**，其本质等同于 `fetch + merge`。
  * 细节：如果不指定远端名称和分支名，则抓取所有并更新当前所在分支。

## 5. 解决远程合并冲突

**场景还原**：
两个开发人员（用户 A 和 用户 B）修改了同一个文件的同一行代码。
1. 用户 A 修改后，先推送到远程仓库。
2. 此时用户 B 在本地也修改了代码，提交到了本地仓库，也需要推送到远程仓库。
3. 因为用户 B 晚于用户 A，远程仓库已经有了 A 的最新提交，所以 B 的推送会被拒绝。

**解决流程**：
1. **拉取代码**：用户 B 必须先执行 `git pull` 拉取远程仓库的最新提交。此时由于修改了同一行，会发生**合并冲突**。
2. **手动解决**：用户 B 打开冲突文件，根据需要将 A 的代码和自己的代码进行合并，并删除 Git 自动产生的冲突标记符。
3. **重新提交与推送**：用户 B 修改完成后，重新执行 `git add` 和 `git commit` 提交到本地仓库，最后再次执行 `git push` 顺利推送到远程分支。
