# Git 基础与分支管理进阶笔记

## 1. Git 环境简介与基本配置

### 1.1 核心工具
* **Git GUI**：Git 官方提供的图形界面工具，方便可视化操作。
* **Git Bash**：Git 官方提供的命令行工具（基于 MinGW64），在 Windows 环境下模拟 Linux 终端，是日常开发中最常用的工具。

### 1.2 全局用户信息配置
在使用 Git 提交代码前，必须设置用户信息，以便在历史记录中追踪是谁做出了修改。
```bash
# 设置用户信息
git config --global user.name "您的用户名"
git config --global user.email "您的邮箱地址"

# 查看配置是否成功
git config --global user.name
git config --global user.email
```

### 1.3 为常用指令配置别名（Alias）
在开发中一些常用命令（如复杂的 log 命令）较长，可以通过配置 Linux 别名来简化输入。
1. **打开用户目录并创建 `.bashrc` 文件**：在终端执行 `touch ~/.bashrc` 或 `code ~/.bashrc`。
2. **在 `.bashrc` 中输入别名配置**（例如将复杂的日志查询简化为 `git-log`）：
   ```bash
   # 用于更加美观和清晰地打印 Git 提交图
   alias git-log='git log --all --pretty=oneline --abbrev-commit --graph'
   ```
3. **刷新配置使其立刻生效**：在 Git Bash 中执行：
   ```bash
   source ~/.bashrc
   ```

## 2. 获取本地仓库与基础操作

### 2.1 初始化本地仓库
1. 在电脑任意位置创建一个空目录作为您的本地仓库。
2. 进入该目录，点击右键选择 **Git Bash Here** 打开窗口。
3. 执行初始化命令：
   ```bash
   git init
   ```
4. 创建成功后，文件夹下会出现一个隐藏的 `.git` 目录（请勿手动修改该目录下的任何文件）。

### 2.2 基础操作指令与文件状态转化
Git 工作目录下的文件修改（增加、删除、更改）会经历不同的状态，通过核心命令进行状态转换：

```text
[工作区 (Workspace)] -- git add --> [暂存区 (Index/Stage)] -- git commit --> [本地仓库 (Repository)]
```

* **查看修改状态 (`git status`)**：查看当前工作区、暂存区的文件状态（如未跟踪、已修改、已暂存）。
* **添加文件到暂存区 (`git add`)**：
  ```bash
  git add 单个文件名        # 暂存指定文件
  git add .               # 暂存当前目录下所有修改及新文件（最常用）
  ```
* **提交文件到本地仓库 (`git commit`)**：
  ```bash
  git commit -m "提交日志注释内容"
  ```

## 3. 版本记录、回退与文件过滤

### 3.1 查看提交日志 (`git log`)
* **基本命令**：`git log [options]`
* **常用参数说明**：
  * `--all`：显示所有分支的提交记录。
  * `--pretty=oneline`：将每次提交的信息精简压缩为一行显示。
  * `--abbrev-commit`：使输出的 `commitID`（散列值）更短、更简洁。
  * `--graph`：以文本图（线段树状图）的形式直观显示分支合并历史。

### 3.2 版本回退与历史抢救
* **版本切换/回退**：
  ```bash
  git reset --hard commitID
  ```
  *(注：`commitID` 可以通过 `git log` 指令查询)*
* **查看被删除的历史记录 (`git reflog`)**：
  如果不小心回退到了旧版本，导致新版本的提交在 `git log` 中看不到了，可以使用 `git reflog` 查找已经被“删除”或隐蔽的提交记录，配合 `git reset` 重新跳转回新版本。

### 3.3 忽略特定文件 (`.gitignore`)
在项目中，有些不必要的系统文件、编译缓存、临时文件或包含敏感信息的配置文件不需要存入仓库。
* **操作方法**：在项目根目录下创建一个名为 `.gitignore` 的文件（名称固定）。
* **配置规则举例**：
  ```text
  *.log          # 忽略所有以 .log 结尾的文件
  target/        # 忽略 target 文件夹及目录下的所有内容
  .idea/         # 忽略 IDE 的配置文件
  ```

## 4. 分支管理与冲突解决

分支允许你从开发主线上分离开来，进行功能开发或 Bug 修复，而不影响主线代码。

### 4.1 分支基础操作

| 核心功能 | 对应命令 | 备注说明 |
| :--- | :--- | :--- |
| **查看本地分支** | `git branch` | 带有 `*` 号的代表当前激活的分支 |
| **创建本地分支** | `git branch 分支名` | 基于当前分支的当前状态切出新分支，绝非从零开始 |
| **切换分支** | `git checkout 分支名` | 切换工作区内容至该分支状态 |
| **创建并切换分支** | `git checkout -b 分支名` | 一键完成创建和切换操作（高频使用） |
| **合并分支** | `git merge 分支名` | 将“目标分支”的代码合并到当前所在的分支上 |
| **安全删除分支** | `git branch -d 分支名` | 删除分支时会执行安全检查，未合并的分支不允许删除 |
| **强制删除分支** | `git branch -D 分支名` | 不做任何检查，直接强制删除分支 |

### 4.2 解决合并冲突 (Conflict)
* **冲突成因**：当两个分支在**同一个文件的同一行**进行了不同的修改，在进行 `git merge` 合并时，Git 无法自动判断保留哪一份，从而报错并中断合并。
* **解决步骤**：
  1. 打开冲突的文件，找到 Git 自动标记的冲突区域（带有 `<<<<<<<`, `=======`, `>>>>>>>` 的部分）。
  2. 留下期望保留的代码，手动删掉 Git 产生的冲突标记符。
  3. 修改完成后，重新执行：
     ```bash
     git add 冲突文件名
     git commit -m "解决冲突"
     ```

### 4.3 合并的快进模式 (Fast-Forward)
* **概念说明**：如果在合并分支时，主分支自切出新分支以来**没有发生过任何新的提交**，那么 Git 在合并时会直接把主分支的指针向前指向新分支的最新提交。这种合并模式称为 **Fast-Forward（快进模式）**。
* **特点**：快进合并不会产生新的合并节点（Merge Commit），其提交历史呈现出一条完美的直线。

## 5. 企业级分支研发原则与流程 (Git Flow)

在团队并行开发中，为了保证线上运行代码的绝对稳定性，通常遵循以下分支管理原则：

```text
  [master (生产)]   ───────►  V1.0  ───────────────────►  V1.1
                       │                               ▲
                       ▼ (派生修复)                     │ (合并上线)
  [hotfix (热修复)]     └──► hotfix-bug ──► [合并返回] ───┤
                       │                               │
                       ▼ (创建主开发)                   │
  [develop (开发)]  ───────────────────► [合并功能] ─────┘
                       │                 ▲
                       ▼ (并行开发)       │ (开发完合并)
  [feature (功能)]     └─► feature-A ────┘
```

1. **`master` (生产分支)**：
   * **定位**：线上生产环境运行的分支，代表最稳定的版本。
   * **原则**：绝对不能直接在 `master` 上编写代码。只有经过严格测试、准备发布的版本，才能由 `develop` 或 `hotfix` 分支合并进来。
2. **`develop` (开发分支)**：
   * **定位**：开发团队的主要演进分支，通常由 `master` 初始化切出。
   * **原则**：承载日常的开发成果。阶段性功能开发完毕并通过团队内测后，最终会合入 `master` 准备上线。
3. **`feature` (功能分支)**：
   * **定位**：同期并行开发、但不同期上线的业务需求/新功能开发分支。
   * **原则**：从 `develop` 分支切出，开发任务完成后通过代码审查（Code Review）合并回 `develop` 分支，随后该 feature 分支通常会被删除。
4. **`hotfix` (紧急修复分支)**：
   * **定位**：专门用来解决线上生产环境（`master`）突发重大 Bug 的突击分支。
   * **原则**：直接从 `master` 分支派生出来进行 Bug 抢救。修复并测试通过后，必须**同时合并回 `master` 和 `develop`（以及当前处于测试中的相关分支）**，确保 Bug 在所有版本中都被根除。
