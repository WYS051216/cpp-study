# 版本控制系统（VCS）知识笔记

## 1. 版本控制的两种主要方式

### 1.1 集中式版本控制工具（Centralized VCS）
* **工作原理**：版本库集中存放在**中央服务器**上。团队中的每个成员在工作时都需要从中央服务器下载最新代码。
* **网络依赖**：**必须联网**才能工作（无论是局域网还是互联网）。个人在本地修改代码后，必须提交回中央版本库。
* **典型代表**：SVN（Subversion）、CVS。

### 1.2 分布式版本控制工具（Distributed VCS）
* **工作原理**：没有绝对的“中央服务器”。**每个开发者的电脑都是一个完整的版本库**。
* **网络依赖**：在本地工作时**不需要互联网**，因为完整的历史记录和版本库都在自己的本地设备上。
* **团队协作**：多人协作时，只需将各自在本地做出的修改推送给对方，或者共同推送到一个共享的远程仓库，即可相互看到并合并修改。
* **典型代表**：Git。

## 2. Git 概述与核心特点

Git 是一款开源的分布式版本控制系统，其核心思想是“去中心化”与“快照存储”。

### 2.1 Git 的主要特点
* **分布式架构**：不依赖中央服务器，每个本地仓库都拥有完整的历史版本信息。即使网络断开或服务器宕机，本地依然可以流畅进行提交、查看历史等所有操作，安全性极高。
* **极高的性能与速度**：由于绝大多数操作（如提交、查看历史、切换分支）都在本地磁盘上直接运行，不需要网络传输，因此执行速度极快。
* **轻量级分支管理**：Git 的分支本质上只是一个指向特定提交对象的指针。创建、切换和合并分支的操作都在瞬间完成，非常适合团队开展并行开发与特性分支管理。
* **数据完整性与安全性**：Git 使用 SHA-1 散列（Hash）算法对所有内容和目录结构进行校验。任何文件在传输或存储过程中发生改变，Git 都能立即察觉，确保历史记录不可篡改。
* **独特的暂存区设计**：在工作区和本地仓库之间引入了**暂存区（Staging Area）**。允许开发者对修改进行精细化筛选，分批、按逻辑组装成不同的提交，使提交历史更加清晰。

## 3. Git 核心工作流程与命令

Git 的日常开发流程主要围绕四个核心区域展开：**工作区（Workspace）**、**暂存区（Index/Stage）**、**本地仓库（Local Repository）**和**远程仓库（Remote Repository）**。

### 3.1 核心命令详解

| 命令 | 英文名称 | 核心作用与数据流向说明 |
| :--- | :--- | :--- |
| **`clone`** | 克隆 | 从远程仓库完整地复制一个代码库到本地，包含所有的文件、历史版本和分支。 |
| **`checkout`** | 检出 | 从本地仓库中检出一个特定的分支或历史版本到工作区，以便进行修订或切换工作上下文。 |
| **`add`** | 添加 | 将工作区中修改或新增的文件提交到**暂存区（Stage）**，标记为待提交状态。 |
| **`commit`** | 提交 | 将暂存区中的所有快照正式提交到**本地仓库**，生成一个新的历史版本节点。 |
| **`fetch`** | 抓取 | 从远程仓库获取最新的版本数据和分支信息到本地仓库，但**不进行任何合并（Merge）**动作，对当前工作区无影响。 |
| **`pull`** | 拉取 | 从远程仓库拉取最新代码并**自动合并**到当前的本地分支和工作区。其本质等同于 `fetch` + `merge`。 |
| **`push`** | 推送 | 将本地仓库中新提交的历史版本推送到远程仓库，实现与团队成员的代码共享。 |
