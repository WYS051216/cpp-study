# 02_GDB 调试器核心笔记

## 1. GDB 调试器简介
GDB (GNU Debugger) 是一个功能强大的 C/C++ 调试器，也是 Linux 系统下开发 C/C++ 后端程序最常用的排错神器。

**GDB 的主要功能：**
* **设置断点**：使程序在指定的代码行、函数或条件上暂停执行，便于观察。
* **单步执行**：一行一行地执行代码，精准定位逻辑错误。
* **查看变量**：在程序暂停时，随时查看或修改程序中变量值的变化。
* **动态环境**：动态改变程序的执行环境。
* **事后诸葛亮**：分析崩溃程序产生的 `core` 内存转储文件，找出导致程序崩溃（如段错误）的元凶。

---

## 2. 调试前的终极前提 (⚠️ 极其重要)
在使用 GDB 调试之前，**编译时必须加上 `-g` 参数**！
如果不加 `-g`，生成的可执行文件中就不会包含源代码的行号、变量名等调试信息，GDB 进去只能看到一堆看不懂的机器码。

* **正确的编译姿势**：
  ```bash
  g++ -g main.cpp -o main
  ```

## 3. GDB 常用命令速查表

进入调试模式：在终端输入 `gdb 可执行文件名`（例如：`gdb main`）。
*(注：GDB 支持命令简写，括号内为最常用的单个字母简写)*

### 🚀 启动与退出
| 命令 | 简写 | 作用说明 |
| :--- | :--- | :--- |
| `run` | `r` | **启动程序**。程序会一直运行，直到遇到第一个断点或程序结束。 |
| `quit` | `q` | **退出 GDB** 调试环境。 |

### 🛑 断点管理
| 命令 | 简写 | 作用说明 |
| :--- | :--- | :--- |
| `break 行号` | `b 15` | 在当前文件的第 15 行**打断点**。 |
| `break 函数名`| `b swap` | 在指定函数的入口处**打断点**。 |
| `info break` | `i b` | **查看**当前设置的所有断点信息（包含断点编号）。 |
| `delete 编号` | `d 1` | **删除**编号为 1 的断点。不加编号则删除所有断点。 |

### 👣 单步执行 (核心操作)
| 命令 | 简写 | 作用说明 |
| :--- | :--- | :--- |
| `next` | `n` | **单步步过**。执行下一行代码；如果遇到函数调用，**不进入**函数内部，直接把函数当做一步执行完。 |
| `step` | `s` | **单步步入**。执行下一行代码；如果遇到函数调用，**会进入**函数内部，追踪里面的细节。 |
| `continue`| `c` | **继续执行**。让暂停的程序继续往下跑，直到遇到下一个断点。 |

### 🔍 变量与代码查看
| 命令 | 简写 | 作用说明 |
| :--- | :--- | :--- |
| `list` | `l` | **查看源代码**。默认每次显示 10 行。继续按回车可往下翻看。 |
| `print 变量` | `p a` | **打印变量** `a` 当前的值。也可以打印表达式，如 `p a+b`。 |
| `display 变量`| `disp a`| **持续跟踪变量**。每次程序暂停或单步执行时，都会自动打印出该变量的值。 |

### 💥 崩溃排查杀手锏
| 命令 | 简写 | 作用说明 |
| :--- | :--- | :--- |
| `backtrace` | `bt` | **查看函数调用栈**。当程序因为指针越界等问题崩溃（段错误）时，输入 `bt` 可以直接查出代码是死在哪一层函数调用的哪一行！ |

## 4. 实战演练：一个标准的排错工作流

假设你的程序 `main` 运行结果不对，你可以按照以下步骤抓虫：

1. **带调试信息编译**：`g++ -g main.cpp -o main`
2. **启动 GDB**：`gdb main`
3. **打断点**：估计错误在第 20 行附近，输入 `b 20`
4. **跑起来**：输入 `r`，程序全速运行，然后在第 20 行停下。
5. **看代码**：输入 `l`，看看第 20 行前后的代码逻辑。
6. **看变量**：输入 `p 变量名`，看看此时的计算结果对不对。
7. **走两步**：输入 `n` 执行下一行，再用 `p` 看看变量变了没有。
8. **揪出 Bug**：发现逻辑不对，修改源代码。
9. **退出**：输入 `q` 退出 GDB，重新编译运行。
