# 设计模式与 UML 笔记整理

---

## 一、设计模式简介

### 1. 什么是设计模式
设计模式是指：

> 在特定环境下，针对反复出现的软件设计问题，所提出的通用、可复用、经过验证的解决方案。

也可以理解为：

> 每个模式都描述了在某种环境中不断出现的问题，以及该问题解决方案的核心思想，从而使我们能够反复使用这些已有的成功经验，而不必重复做相同的工作。

---

### 2. 设计模式的基本要素
一个设计模式通常包含以下几个部分：

#### （1）模式名称
用一两个词来概括模式的问题或解决方案，便于记忆和交流。

#### （2）问题
说明该模式适用于什么场景，描述设计中存在的问题及产生问题的原因。

#### （3）目的
说明该模式想要达到的目标，也就是为什么要使用这个模式。

#### （4）解决方案
描述模式的组成部分、各部分之间的关系、职责及协作方式。  
通常会借助 **UML 类图** 和 **核心代码** 进行说明。

#### （5）效果
描述模式的优点、缺点，以及在使用时需要权衡的问题。

---

## 二、设计模式的分类

### 1. 按用途分类

#### （1）创建型模式
主要用于描述 **如何创建对象**。

#### （2）结构型模式
主要用于描述 **如何组织类和对象之间的关系**，形成更灵活的结构。

#### （3）行为型模式
主要用于描述 **类或对象之间如何交互，以及如何分配职责**。

---

### 2. 按范围分类

#### （1）类模式
处理 **类与子类之间的关系**，主要通过 **继承** 来实现。  
这种关系在 **编译时就已经确定**，属于静态关系。

#### （2）对象模式
处理 **对象之间的关系**，这种关系在 **运行时可以变化**，动态性更强。

---

## 三、GoF 23 种设计模式

GoF（四人帮）将设计模式分为三大类，共 23 种。

### 1. 创建型模式（5种）
- 单例模式（Singleton）
- 工厂方法模式（Factory Method）
- 抽象工厂模式（Abstract Factory）
- 建造者模式（Builder）
- 原型模式（Prototype）

### 2. 结构型模式（7种）
- 适配器模式（Adapter）
- 桥接模式（Bridge）
- 组合模式（Composite）
- 装饰模式（Decorator）
- 外观模式（Facade）
- 享元模式（Flyweight）
- 代理模式（Proxy）

### 3. 行为型模式（11种）
- 职责链模式（Chain of Responsibility）
- 命令模式（Command）
- 解释器模式（Interpreter）
- 迭代器模式（Iterator）
- 中介者模式（Mediator）
- 备忘录模式（Memento）
- 观察者模式（Observer）
- 状态模式（State）
- 策略模式（Strategy）
- 模板方法模式（Template Method）
- 访问者模式（Visitor）

---


## 四、设计模式的优点

1. **提高代码复用性和可维护性**，提高开发效率与设计效率。  
2. 提供一套 **通用的设计词汇和表达方式**，便于开发人员之间沟通交流。  
3. **提高系统的灵活性和可扩展性**。  
4. 有助于他人 **更快理解系统结构与设计思想**。  
5. 有助于初学者 **更深入地理解面向对象思想**。

# 五、UML 简介

## 1. UML 概述
UML（Unified Modeling Language，统一建模语言）是一种面向对象的、开放的建模语言，用于：

- 说明系统
- 可视化系统
- 构建系统
- 编写系统文档

UML 总结了许多被实践验证有效的工程经验，特别适用于 **大型、复杂、软件密集型系统** 的建模。

---

## 2. UML 系统开发中的三种主要模型

### （1）功能模型

功能模型是从**用户角度**展示系统的功能，关注的是：

> 系统可以为用户提供哪些功能。

功能模型通常使用 **用例图** 来描述。

#### 例子：网上书店系统的功能模型

用户在网上书店系统中可以完成以下操作：

- 注册/登录
- 浏览图书
- 搜索图书
- 下单购买
- 支付订单
- 查看订单

#### 图示（功能模型 / 用例图示意）

```text
           +----------------------+
           |     网上书店系统      |
           |----------------------|
用户 -----> | 注册/登录             |
用户 -----> | 浏览图书              |
用户 -----> | 搜索图书              |
用户 -----> | 下单购买              |
用户 -----> | 支付订单              |
用户 -----> | 查看订单              |
           +----------------------+
```

#### 说明

这个模型不关心系统内部是怎么实现的，只关心：

- 用户是谁
- 用户能做什么
- 系统对外提供了哪些功能

---

### （2）对象模型

对象模型通过对象、属性、操作、关联等概念来描述系统的**静态结构**。

对象模型通常包括：

- 类图
- 对象图

它关注的是：

> 系统由哪些对象或类组成，它们之间是什么关系。

#### 例子：网上书店系统的对象模型

在网上书店系统中，可以抽象出以下几个类：

- 用户（User）
- 图书（Book）
- 订单（Order）
- 订单项（OrderItem）

它们之间的关系如下：

- 一个用户可以拥有多个订单
- 一个订单可以包含多个订单项
- 一个订单项对应一本图书

#### 图示（对象模型 / 类图示意）

```text
+----------------+        1        *       +----------------+
|      User      |------------------------>|     Order      |
+----------------+                         +----------------+
| - userId       |                         | - orderId       |
| - name         |                         | - createTime    |
+----------------+                         +----------------+
                                                  |
                                                  | 1
                                                  |
                                                  | *
                                           +----------------+
                                           |   OrderItem    |
                                           +----------------+
                                           | - count        |
                                           | - price        |
                                           +----------------+
                                                  |
                                                  | *
                                                  | 1
                                           +----------------+
                                           |      Book      |
                                           +----------------+
                                           | - bookId       |
                                           | - title        |
                                           | - price        |
                                           +----------------+
```

#### 说明

对象模型主要描述：

- 系统中有哪些类
- 每个类有哪些属性和操作
- 类与类之间如何关联

也就是说，它反映的是系统的**结构**。

---

### （3）动态模型

动态模型用于描述系统的**动态行为和运行过程**。

它关注的是：

> 系统在运行过程中，对象之间如何交互，流程如何推进，状态如何变化。

动态模型通常包括：

- 序列图（时序图）
- 活动图
- 状态图

---

#### 1）序列图示例

##### 场景：用户在网上书店下单购买图书

```text
用户        订单页面        订单服务        库存服务        支付服务
 |              |              |              |              |
 |--提交订单---->|              |              |              |
 |              |--创建订单---->|              |              |
 |              |              |--检查库存---->|              |
 |              |              |<---库存足够---|              |
 |              |              |--生成订单---->|              |
 |              |<---返回订单---|              |              |
 |--发起支付----------------------------------------------->|
 |<-------------------------支付成功------------------------|
```

##### 说明

序列图强调的是：

- 对象之间谁调用谁
- 调用顺序是什么
- 系统是怎样一步一步执行的

---

#### 2）活动图示例

##### 场景：用户购买图书的业务流程

```text
开始
  |
  v
浏览图书
  |
  v
选择图书
  |
  v
加入购物车
  |
  v
提交订单
  |
  v
支付订单
  |
  v
支付成功？
 / \
否  是
|    |
v    v
结束 生成订单
       |
       v
      结束
```

##### 说明

活动图强调的是：

- 业务流程如何流转
- 每一步操作的执行顺序
- 是否存在条件分支

适合用来描述完整的业务过程。

---

#### 3）状态图示例

##### 场景：订单状态变化过程

```text
[新建订单] --> [待支付] --> [已支付] --> [已发货] --> [已完成]
                  |
                  v
               [已取消]
```

##### 说明

状态图强调的是：

- 某个对象有哪些状态
- 状态之间如何转换
- 在什么条件下进入下一个状态

例如订单对象在运行过程中，可能会依次经历：

- 新建订单
- 待支付
- 已支付
- 已发货
- 已完成
- 已取消

---

#### 总结

三种模型的区别可以概括为：

| 模型 | 关注点 | 常见图 | 举例 |
|------|--------|--------|------|
| 功能模型 | 系统能做什么 | 用例图 | 用户登录、搜索图书、下单 |
| 对象模型 | 系统由什么组成 | 类图、对象图 | User、Book、Order 之间的关系 |
| 动态模型 | 系统怎么运行 | 序列图、活动图、状态图 | 下单流程、支付流程、订单状态变化 |

#### 记忆方法

- **功能模型**：从用户角度看，关注“做什么”
- **对象模型**：从结构角度看，关注“有什么”
- **动态模型**：从运行角度看，关注“怎么做”

## 3. UML 图
要区分 **UML 模型** 和 **UML 图**：

- **UML 模型**：对系统的抽象描述
- **UML 图**：模型信息的图形化表达方式

UML 中一共定义了 **14 种图**。

# 六、类图简介

在 UML 中，类图一般由三部分组成：

## 1. 类名
按照命名规范定义，通常写在类图最上方。

## 2. 属性
属性通常表示类的成员变量。  
定义格式：

```text
[可见性] 属性名: 数据类型 [= 默认值]
```

例如：

```text
- name: string
- age: int = 18
```

## 3. 操作
操作通常表示类的成员方法。  
定义格式：

```text
[可见性] 方法名([参数1: 数据类型, 参数2: 数据类型...]) [: 返回值类型]
```

例如：

```text
+ getName(): string
+ setName(name: string): void
```

# 七、类之间的关系

## 1. 关联关系（Association）

### 概念
关联关系是一种结构化关系，用于表示一个类对象与另一个类对象之间存在联系。  
在代码中，通常通过 **成员变量** 来体现。  
在 UML 中，关联关系用 **带箭头或不带箭头的实线** 表示。

关联关系可分为：
- 单向关联
- 双向关联
- 自关联

### （1）单向关联
一个类知道另一个类，但另一个类不一定知道前者。

#### C++ 代码示例
```cpp
#include <iostream>
#include <string>
using namespace std;

class Address {
private:
    string city;

public:

//使用初始化列表的构造方式，专门用于在对象生命周期的最开始阶段为成员变量赋值
    Address(const string& city) : city(city) {}

//在成员函数后面加const后，这个函数是一个只读的状态，不会修改类中的成员变量
    string getCity() const {
        return city;
    }
};

class Person {
private:
    string name;
    Address* address;  // Person 单向关联 Address

public:
    Person(const string& name, Address* address) : name(name), address(address) {}

    void display() const {
        cout << name << " 住在 " << address->getCity() << endl;
    }
};

int main() {
    Address addr("北京");
    Person p("张三", &addr);
    p.display();
    return 0;
}
```

### （2）双向关联
两个类互相知道对方。

#### C++ 代码示例
```cpp
#include <iostream>
#include <string>
using namespace std;

class Wife;  // 前向声明

class Husband {
private:
    string name;
    Wife* wife;

public:
//在双向关联中，不能直接在初始化列表中把他们设置为值，此处的第二个参数的构造虽然说可以删除，但是为了防止随机的野指针，最好别删。
    Husband(const string& name) : name(name), wife(nullptr) {}

    void setWife(Wife* wife) {
        this->wife = wife;
    }

    string getName() const {
        return name;
    }
};

class Wife {
private:
    string name;
    Husband* husband;

public:
    Wife(const string& name) : name(name), husband(nullptr) {}

    void setHusband(Husband* husband) {
        this->husband = husband;
    }

    void introduce() const {
        if (husband != nullptr) {
            cout << "丈夫是：" << husband->getName() << endl;
        }
    }
};

int main() {
    Husband h("李强");
    Wife w("小美");

    h.setWife(&w);
    w.setHusband(&h);

    w.introduce();
    return 0;
}
```

### （3）自关联
一个类中的对象与自身类的其他对象发生关联。

#### C++ 代码示例
```cpp
#include <iostream>
#include <string>
using namespace std;

class Employee {
private:
    string name;
    Employee* manager;  // 自关联：员工的领导也是员工

public:
    Employee(const string& name) : name(name), manager(nullptr) {}

    void setManager(Employee* manager) {
        this->manager = manager;
    }

    void showManager() const {
        if (manager != nullptr) {
            cout << name << " 的领导是 " << manager->name << endl;
        }
    }
};

int main() {
    Employee leader("王经理");
    Employee staff("小张");

    staff.setManager(&leader);
    staff.showManager();

    return 0;
}
```

## 2. 继承关系（Generalization）

### 概念
继承关系也称为 **泛化关系**，用于描述父类与子类之间的关系。  

- 父类：基类、超类
- 子类：派生类

在 UML 中，继承关系用 **空心三角形 + 实线** 表示，箭头指向父类。

### （1）普通继承关系

#### C++ 代码示例
```cpp
#include <iostream>
using namespace std;

class Animal {
public:
    void eat() {
        cout << "动物在吃东西" << endl;
    }
};

class Dog : public Animal {
public:
    void bark() {
        cout << "狗在叫" << endl;
    }
};

int main() {
    Dog dog;
    dog.eat();
    dog.bark();
    return 0;
}
```

### （2）抽象继承关系

#### C++ 代码示例
```cpp
#include <iostream>
using namespace std;

class Shape {
public:
    virtual void draw() = 0;  // 纯虚函数
    virtual ~Shape() = default;
};

class Circle : public Shape {
public:
    void draw() override {
        cout << "绘制圆形" << endl;
    }
};

int main() {
    Shape* shape = new Circle();
    shape->draw();

    delete shape;
    return 0;
}
```

## 3. 聚合关系（Aggregation）

### 概念
聚合表示 **整体与部分** 的关系。  
在聚合关系中，部分对象是整体的一部分，但 **可以脱离整体独立存在**。  

在 UML 中，聚合关系用 **空心菱形 + 实线** 表示。

例如：部门与员工。部门没了，员工依然可以存在。

#### C++ 代码示例
```cpp
#include <iostream>
#include <string>
using namespace std;

class Employee {
private:
    string name;

public:
    Employee(const string& name) : name(name) {}

    string getName() const {
        return name;
    }
};

class Department {
private:
    string deptName;
    Employee* employee;  // 聚合关系：员工可独立存在

public:
    Department(const string& deptName, Employee* employee)
        : deptName(deptName), employee(employee) {}

    void show() const {
        cout << deptName << " 部门有员工：" << employee->getName() << endl;
    }
};

int main() {
    Employee emp("张三");
    Department dept("技术部", &emp);

    dept.show();
    return 0;
}
```

---

## 4. 组合关系（Composition）

### 概念
组合也表示 **整体与部分** 的关系，但它比聚合更强。  

在组合关系中：
- 整体对象控制部分对象的生命周期
- 一旦整体不存在，部分通常也不存在
- 整体与部分具有“同生共死”的特点

在 UML 中，组合关系用 **实心菱形 + 实线** 表示。

例如：头与人，房间与房子。

#### C++ 代码示例
```cpp
#include <iostream>
using namespace std;

class Heart {
public:
    void beat() const {
        cout << "心脏在跳动" << endl;
    }
};

class Person {
private:
    Heart heart;  // 组合关系：Heart 作为 Person 的组成部分

public:
    void live() const {
        heart.beat();
        cout << "人活着" << endl;
    }
};

int main() {
    Person p;
    p.live();
    return 0;
}
```

## 5. 依赖关系（Dependency）

### 概念
依赖关系是一种 **使用关系**。  
如果一个类的变化可能影响另一个类，就说明它们之间存在依赖关系。

在 UML 中，依赖关系用 **带箭头的虚线** 表示，箭头从依赖方指向被依赖方。

依赖关系通常通过以下三种方式实现：
1. 将一个类的对象作为另一个类方法的参数  
2. 在一个类的方法中将另一个类的对象作为局部变量  
3. 在一个类的方法中调用另一个类的静态方法  

### （1）作为方法参数

#### C++ 代码示例
```cpp
#include <iostream>
using namespace std;

class Book {
public:
    void read() const {
        cout << "读书" << endl;
    }
};

class Student {
public:
    void study(const Book& book) {  // Book 作为参数，形成依赖关系
        book.read();
    }
};

int main() {
    Book book;
    Student stu;
    stu.study(book);
    return 0;
}
```

### （2）作为局部变量

#### C++ 代码示例
```cpp
#include <iostream>
#include <string>
using namespace std;

class Printer {
public:
    void print(const string& msg) const {
        cout << msg << endl;
    }
};

class Report {
public:
    void createReport() {
        Printer printer;  // 局部变量形式依赖 Printer
        printer.print("生成报表");
    }
};

int main() {
    Report report;
    report.createReport();
    return 0;
}
```

### （3）调用静态方法

#### C++ 代码示例
```cpp
#include <iostream>
using namespace std;

class MathUtil {
public:
    static int add(int a, int b) {
        return a + b;
    }
};

class Calculator {
public:
    void count() const {
        int result = MathUtil::add(3, 5);  // 依赖 MathUtil
        cout << "结果：" << result << endl;
    }
};

int main() {
    Calculator c;
    c.count();
    return 0;
}
```

# 八、总结

## 1. 设计模式核心记忆
- 设计模式是对常见软件设计问题的成熟解决方案
- 学习设计模式有助于提高复用性、可维护性、扩展性和沟通效率
- GoF 23 种模式分为：创建型、结构型、行为型

## 2. UML 核心记忆
- UML 是统一建模语言，用于描述和可视化面向对象系统
- UML 开发中主要包括：功能模型、对象模型、动态模型
- 类图重点掌握：类的组成 + 类之间的关系

## 3. 类之间关系重点区分
- **关联**：有联系，通常体现在成员变量
- **继承**：is-a 关系
- **聚合**：整体包含部分，但部分可独立存在
- **组合**：整体包含部分，部分不能独立存在
- **依赖**：临时使用关系，常体现在方法参数、局部变量或静态调用
