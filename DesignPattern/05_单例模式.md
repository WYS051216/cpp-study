# 单例模式 (Singleton Pattern) 深度解析

## 1. 单例模式的概念
单例模式是一种**对象创建型**设计模式。
在一个项目中，某个类在全局范围内应该**只存在一个实例**。程序通过这个唯一实例向其他模块提供服务。

**典型应用场景：**
* 配置管理器 (ConfigManager)
* 日志管理器 (Logger)
* 线程池 (ThreadPool)
* 数据库连接池
* 全局资源管理对象

---

## 2. 创建对象涉及的常见操作
一个普通的类可以通过以下路径产生多个对象。要实现单例，必须封死这些路径：
1. **默认构造函数**：`Singleton obj;`
2. **拷贝构造函数**：`Singleton obj2(obj1);`
3. **拷贝赋值运算符**：`obj2 = obj1;`

---

## 3. 实现步骤与核心逻辑

### 3.1 构造函数私有化
将构造函数设为 `private`，切断类外部创建对象的权利。
```cpp
private:
    Singleton() {}
```

### 3.2 禁用拷贝与赋值 (C++11 标准)
防止通过已有对象克隆出新对象。
```cpp
public:
    Singleton(const Singleton&) = delete;            // 禁止拷贝构造
    Singleton& operator=(const Singleton&) = delete; // 禁止拷贝赋值
```

### 3.3 提供全局访问接口
通过静态成员函数返回唯一的实例。
```cpp
public:
    static Singleton& getInstance() {
        static Singleton instance; // 局部静态变量
        return instance;
    }
```

---

## 4. 饿汉式 vs 懒汉式

### 4.1 饿汉式 (Eager Initialization)
**特点：** 类加载时（程序启动）就立即创建实例。
* **优点：** 线程安全，访问速度快。
* **缺点：** 如果该实例一直未被使用，会白白占用内存资源。

### 4.2 懒汉式 (Lazy Initialization)
**特点：** 只有在第一次调用 `getInstance()` 时才创建实例（即“延迟加载”）。
* **优点：** 节省内存。
* **缺点：** 在多线程环境下需要处理竞态条件（线程安全问题）。



---

## 5. 线程安全的单例 (Meyers' Singleton)
在 C++11 之后，最推荐的写法是使用**局部静态变量**。这种写法被称为“迈耶斯单例”：

```cpp
static Singleton& getInstance() {
    // C++11 标准保证：如果多个线程同时初始化局部静态变量，
    // 系统会阻塞其他线程，确保初始化只发生一次。
    static Singleton instance; 
    return instance;
}
```
**优势：**
1. **线程安全**：由编译器底层保证。
2. **延迟加载**：第一次调用时才初始化。
3. **自动销毁**：程序结束时自动调用析构函数，无内存泄漏风险。

## 6.1 完整示例代码

```cpp
#include<iostream>
using namespace std;

//饿汉模式->定义类的时候创建单例对象
//定义一个任务队列
class TaskQueue
{
public:
    TaskQueue(const TaskQueue& t) = delete;
    TaskQueue& operator = (const TaskQueue& t) = delete;
    static TaskQueue* getInstance()
    {
        return m_taskQ;
    }

    void print()
    {
        cout << "我是单例对象的一个成员函数..." << endl;
    }

private:
    TaskQueue() = default;
    // TaskQueue(const TaskQueue& t) = default;
    // TaskQueue& operator = (const TaskQueue& t) = default;
    //只能通过类名访问静态属性或方法
    static TaskQueue* m_taskQ;
};

TaskQueue* TaskQueue::m_taskQ = new TaskQueue;

int main()
{
    TaskQueue* taskQ = TaskQueue::getInstance();
    taskQ->print();
    return 0;
}
```

## 6.2 完整实例代码
```cpp
#include<iostream>
#include<mutex>
using namespace std;

//懒汉模式->什么时候使用这个单例对象，在使用的时候再去创建对应的实例
class TaskQueue
{
public:
    TaskQueue(const TaskQueue& t) = delete;
    TaskQueue& operator = (const TaskQueue& t) = delete;
    static TaskQueue* getInstance()
    {
        if(m_taskQ == nullptr)
        {
            m_mutex.lock();
            if(m_taskQ == nullptr)
            {
                m_taskQ = new TaskQueue;
            }
            m_mutex.unlock();
        }
        return m_taskQ;
    }

    void print()
    {
        cout << "我是单例对象的一个成员函数..." << endl;
    }

private:
    TaskQueue() = default;
    // TaskQueue(const TaskQueue& t) = default;
    // TaskQueue& operator = (const TaskQueue& t) = default;
    //只能通过类名访问静态属性或方法
    static TaskQueue* m_taskQ;
    static mutex m_mutex;
};

TaskQueue* TaskQueue::m_taskQ = nullptr;
mutex TaskQueue::m_mutex;

int main()
{
    TaskQueue* taskQ = TaskQueue::getInstance();
    taskQ->print();
    return 0;
}
```

## 6.3 双重检查存在的问题：指令重排序问题
### 6.3.1 问题的根源
在执行 `m_taskQ = new TaskQueue;` 时，编译器和 CPU 为了优化性能，可能会对指令进行重排序。
`new` 操作在底层通常分为三步：
1. **分配内存**：分配一块内存空间。
2. **调用构造函数**：在内存上初始化对象数据。
3. **赋值指针**：将 `m_taskQ` 指向这块内存。

**乱序风险**：如果顺序变成了 **1 -> 3 -> 2**，当执行完第 3 步但尚未执行第 2 步时，`m_taskQ` 已经不是 `nullptr` 了。此时若有另一个线程进入，会拿到一个**尚未构造完成的“半成品”对象**，从而导致程序崩溃。

---

## 6.3.2 进阶方案：使用原子变量 (std::atomic)

为了解决指令重排序问题，现代 C++ 推荐使用 `std::atomic`。它通过**内存屏障 (Memory Barrier)** 确保：指针的赋值操作一定发生在对象构造完成之后。

### 6.3.3 原子变量版示例代码

```cpp
#include<iostream>
#include<mutex>
#include<atomic>
using namespace std;

//懒汉模式->什么时候使用这个单例对象，在使用的时候再去创建对应的实例
class TaskQueue
{
public:
    TaskQueue(const TaskQueue& t) = delete;
    TaskQueue& operator = (const TaskQueue& t) = delete;
    static TaskQueue* getInstance()
    {
        TaskQueue* task = m_taskQ.load();
        if(task == nullptr)
        {
            m_mutex.lock();
            task = m_taskQ.load();
            if(task == nullptr)
            {
                task = new TaskQueue;
                m_taskQ.store(task);
            }
            m_mutex.unlock();
        }
        return task;
    }

    void print()
    {
        cout << "我是单例对象的一个成员函数..." << endl;
    }

private:
    TaskQueue() = default;
    // TaskQueue(const TaskQueue& t) = default;
    // TaskQueue& operator = (const TaskQueue& t) = default;
    //只能通过类名访问静态属性或方法
    //static TaskQueue* m_taskQ;
    static mutex m_mutex;
    static atomic<TaskQueue*> m_taskQ;
};

//TaskQueue* TaskQueue::m_taskQ = nullptr;
atomic<TaskQueue*> TaskQueue::m_taskQ;
mutex TaskQueue::m_mutex;

int main()
{
    TaskQueue* taskQ = TaskQueue::getInstance();
    taskQ->print();
    return 0;
}
```

## 6.3.4 终极推荐：C++11 局部静态变量 (Meyers' Singleton)

如果你使用的是 C++11 或更高标准，其实不需要手动写原子变量或双重检查。

```cpp
#include<iostream>
#include<mutex>
#include<atomic>
using namespace std;

//懒汉模式->什么时候使用这个单例对象，在使用的时候再去创建对应的实例
class TaskQueue
{
public:
    TaskQueue(const TaskQueue& t) = delete;
    TaskQueue& operator = (const TaskQueue& t) = delete;
    static TaskQueue* getInstance()
    {
        //如果多个线程同时试图初始化同一个静态局部变量，初始化过程必须是线程安全的
        static TaskQueue task;
        return &task;
    }

    void print()
    {
        cout << "我是单例对象的一个成员函数..." << endl;
    }

private:
    TaskQueue() = default;
    // TaskQueue(const TaskQueue& t) = default;
    // TaskQueue& operator = (const TaskQueue& t) = default;
};

int main()
{
    TaskQueue* taskQ = TaskQueue::getInstance();
    taskQ->print();
    return 0;
}
```
**优点：** 代码最简洁、性能高、天然支持线程安全，且不存在指令重排序导致的半成品对象问题。

## 6.3.5 总结对照表

| 实现方式 | 优点 | 缺点 |
| :--- | :--- | :--- |
| **饿汉式** | 绝对线程安全，访问快 | 启动慢，不使用也占内存 |
| **基础懒汉式** | 节省内存 | 多线程不安全 |
| **双重检查 (DCLP)** | 兼顾性能与安全 | 存在指令重排序风险 |
| **原子变量版** | 解决重排序，安全可靠 | 代码较复杂 |
| **局部静态变量** | **最推荐**：简洁、安全、高性能 | C++11 之前不支持线程安全 |

## 7. 核心思想总结
* **禁生**：构造函数私有，防止外部 new。
* **禁改**：禁用拷贝和赋值。
* **常驻**：使用 `static` 保证实例的生命周期和全局访问性。
* **安全**：利用 C++11 局部静态变量特性保证多线程环境下的唯一性。
