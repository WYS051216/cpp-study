# 设计模式核心原则笔记

设计模式的本质是为了提高代码的**灵活性、可维护性**和**复用性**。以下是其遵循的四大核心原则：

---

## 1. 单一职责原则 (Single Responsibility Principle, SRP)

### 核心描述
**“对一个类而言，应该只有一个引起它变化的原因。”**

* **本质：** 将类承担的职责单一化。
* **细节补充：**
    * 如果一个类承担了过多的职责，这些职责就会耦合在一起。当其中一个职责发生变化时，可能会削弱或破坏该类完成其他职责的能力。
    * **优点：** 降低类的复杂度，提高代码的可读性和可维护性。

---

## 2. 开放-封闭原则 (Open-Closed Principle, OCP)

### 核心描述
**“对扩展开放，对修改封闭。”**

* **开放：** 模块、类、函数等应该是可以扩展的（通过继承、多态等方式增加新功能）。
* **封闭：** 在增加新功能时，不应该修改已经存在的源代码（保持核心逻辑的稳定性）。
* **细节补充：**
    * **实现手段：** 抽象是关键。通过定义一个固定的抽象层，将具体的实现留给派生类。
    * **应用场景：** 当需求变化时，我们通过添加新的子类来适应变化，而不是去改动原有的业务逻辑代码。

---

## 3. 依赖倒置原则 (Dependency Inversion Principle, DIP)

### 核心描述
**“高层模块不应该依赖低层模块，两者都应该依赖抽象；抽象不应该依赖细节，细节应该依赖抽象。”**

* **层级概念：**
    1.  **高层模块：** 业务逻辑层。负责处理复杂的业务流程。
    2.  **低层模块：** 基础库/工具层。如 API 封装、数据库操作、动态库等。
    3.  **抽象：** C++ 中的**抽象类**（纯虚函数声明）。
* **细节补充：**
    * **传统做法：** 高层直接调用低层。缺点是低层一变，高层全崩，耦合度极高。
    * **倒置做法：** 高层定义一个抽象接口，低层去实现这个接口。高层只针对接口编程，不再关心低层的具体实现细节。
    * **C++ 特点：** 在 C++ 中，这种原则主要通过包含纯虚函数的抽象基类来实现。

---

## 4. 里氏代换原则 (Liskov Substitution Principle, LSP)

### 核心描述
**“子类类型必须能够替换掉它们的父类类型。”**

* **本质：** 确保继承关系的正确性。
* **细节补充：**
    * **要求：** 在任何父类出现的地方，子类对象都可以完美替代，且程序行为不发生改变。
    * **意义：** 它是实现“开放-封闭原则”的重要基础。只有当子类能完全替代父类时，我们才能在不修改高层代码的情况下，通过传入不同的子类来扩展功能。
    * **约束：** 子类可以实现父类的抽象方法，但不能随意修改父类中已经实现好的方法逻辑。

---

## 💡 总结对照表

| 原则名称 | 核心目标 | 关键手段 |
| :--- | :--- | :--- |
| **单一职责** | 降低类之间的耦合 | 职能拆分 |
| **开放封闭** | 增加系统稳定性 | 抽象化、多态 |
| **依赖倒置** | 解耦高低层逻辑 | 针对接口编程 |
| **里氏代换** | 保证继承的健壮性 | 子类完全兼容父类行为 |
